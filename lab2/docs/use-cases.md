# Use Cases: Autistic Bank API

## Реєстрація користувача

**Актор:** Неавторизований користувач

**Передумови:** Відсутні

**Основний сценарій:**
1. Гість відправляє `POST`-запит з email, паролем та ПІБ.
2. Система валідує формат email (інваріант: має містити `@`).
3. Система перевіряє що пароль не менше 6 символів (інваріант).
4. Система перевіряє що email не зайнятий в базі даних.
5. Система хешує пароль через BCrypt та створює запис користувача.
6. Система повертає статус `201 Created` та JWT-токен доступу.

**Альтернативні сценарії / помилки:**
* Невалідний формат email або відсутній `@`: `400 Bad Request`
* Пароль коротший за 6 символів: `400 Bad Request`
* Користувач з таким email вже існує: `409 Conflict`

## Вхід користувача

**Актор:** Неавторизований користувач

**Передумови:** Користувач має існуючий акаунт у системі

**Основний сценарій:**
1. Користувач відправляє `POST`-запит з email та паролем.
2. Система знаходить користувача за email у базі даних.
3. Система порівнює отриманий пароль з BCrypt хешем у базі.
4. При успішному збігу система генерує JWT-токен.
5. Система повертає статус `200 OK` та токен доступу.

**Альтернативні сценарії / помилки:**
* Email не знайдено в базі: `401 Unauthorized`
* Пароль не співпадає з хешем: `401 Unauthorized`
* Відсутні обов'язкові поля: `400 Bad Request`

## Створення банківського рахунку

**Актор:** Авторизований користувач

**Передумови:** Користувач успішно автентифікований та має дійсний JWT-токен

**Основний сценарій:**
1. Користувач відправляє `POST`-запит із зазначенням валюти (USD, UAH, EUR).
2. Система перевіряє та розпаковує JWT-токен, отримує ID користувача.
3. Система перевіряє що валюта входить до списку дозволених (інваріант).
4. Система генерує унікальний номер рахунку та встановлює початковий баланс 0.
5. Система зберігає рахунок та повертає статус `201 Created` з даними рахунку.

**Альтернативні сценарії / помилки:**
* Відсутній або невалідний JWT-токен: `401 Unauthorized`
* Валюта не входить до списку дозволених: `400 Bad Request`

## Перегляд рахунків користувача

**Актор:** Авторизований користувач

**Передумови:** Користувач успішно автентифікований

**Основний сценарій:**
1. Користувач відправляє `GET`-запит.
2. Система перевіряє JWT-токен та отримує ID користувача.
3. Система вибирає всі рахунки що належать поточному користувачу.
4. Система повертає статус `200 OK` та список рахунків.

**Альтернативні сценарії / помилки:**
* Відсутній або невалідний JWT-токен: `401 Unauthorized`

## Переказ коштів між рахунками

**Актор:** Авторизований користувач

**Передумови:** Користувач має рахунок відправника з достатнім балансом

**Основний сценарій:**
1. Користувач відправляє `POST`-запит з ID рахунку відправника, ID рахунку отримувача та сумою.
2. Система перевіряє JWT-токен та отримує ID користувача.
3. Система перевіряє що рахунки відправника та отримувача різні (інваріант).
4. Система знаходить рахунок відправника та перевіряє що він належить поточному користувачу.
5. Система знаходить рахунок отримувача.
6. Система перевіряє інваріант: сума переказу > 0.
7. Система перевіряє інваріант: баланс відправника >= сума переказу.
8. Система знімає кошти з рахунку відправника та зараховує отримувачу.
9. Система створює запис транзакції в базі даних.
10. Система повертає статус `200 OK`.

**Альтернативні сценарії / помилки:**
* Відсутній або невалідний JWT-токен: `401 Unauthorized`
* ID рахунків відправника та отримувача співпадають: `400 Bad Request`
* Сума переказу <= 0: `400 Bad Request`
* Рахунок відправника або отримувача не знайдено: `404 Not Found`
* Недостатньо коштів на рахунку: `400 Bad Request`

## Поповнення рахунку

**Актор:** Авторизований користувач

**Передумови:** Користувач має існуючий рахунок

**Основний сценарій:**
1. Користувач відправляє `POST`-запит з ID рахунку та сумою поповнення.
2. Система перевіряє JWT-токен та отримує ID користувача.
3. Система знаходить рахунок та перевіряє що він належить поточному користувачу.
4. Система перевіряє інваріант: сума > 0.
5. Система додає кошти до балансу рахунку.
6. Система повертає статус `200 OK`.

**Альтернативні сценарії / помилки:**
* Відсутній або невалідний JWT-токен: `401 Unauthorized`
* Рахунок не знайдено або не належить користувачу: `404 Not Found`
* Сума <= 0: `400 Bad Request`

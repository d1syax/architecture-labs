# ADR: Вибір доменної моделі

## Статус
Прийнято

## Контекст
При рефакторингу лабораторної роботи 1 необхідно обрати підхід до доменної моделі:
Anemic Domain Model або Rich Domain Model.

## Рішення
Обрано **Rich Domain Model**.

## Обґрунтування
В Anemic Domain Model сутності - це просто мішки з даними, а вся логіка
знаходиться в сервісах. Це призводить до того, що інваріанти не захищені
самою сутністю - їх можна порушити звернувшись до властивостей напряму.

В Rich Domain Model логіка інкапсульована всередині сутностей:
- `Account.Debit()` сам перевіряє баланс і повертає `DomainError`
- `Account.Credit()` сам перевіряє що сума більше нуля
- `User.VerifyPassword()` інкапсулює логіку перевірки пароля
- Всі властивості мають `private set` — стан не можна змінити обходячи методи

## Наслідки
Переваги:
- Інваріанти захищені на рівні самої сутності
- Доменна логіка не дублюється в сервісах
- Unit тести домену не потребують жодної інфраструктури

Недоліки:
- Складніший мапінг між доменними моделями та ORM entities
- Більше коду порівняно з Anemic підходом

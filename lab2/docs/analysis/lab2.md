# Аналіз: Лабораторна робота 2

## Що змінилося в структурі проєкту порівняно з лабораторною 1?

В лабораторній роботі 1 весь код знаходився в одному проєкті `MyBank.Api`
з папками `Controllers`, `Services`, `Models`, `Data`. Всі шари були фізично
в одному місці і нічого не заважало сервісу звернутись до будь-якого класу.

В лабораторній роботі 2 проєкт розділено на 4 окремих `csproj`:
- `MyBank.Domain` — доменні моделі, інтерфейси репозиторіїв, доменні помилки
- `MyBank.Application` — сервіси з бізнес-логікою, інтерфейс токен-сервісу
- `MyBank.Infrastructure` — реалізація репозиторіїв, EF Core entities, маппери, JWT
- `MyBank.Api` — контролери, DTO

Залежності між шарами тепер фізично обмежені через `ProjectReference`.
`MyBank.Domain` не має жодних зовнішніх залежностей — він не знає
про EF Core, ASP.NET або JWT. Це гарантується самою структурою проєкту.

## Які переваги дало розділення на шари?

Головна перевага — інверсія залежностей. В лабі 1 `AccountService` напряму
залежав від `BankDbContext`. В лабі 2 `AccountService` залежить від
`IAccountRepository` — інтерфейсу який визначений в домені. Реальна
реалізація через EF Core підставляється через DI контейнер.

Це дало три конкретні переваги. По-перше, unit тести домену запускаються
без жодної бази даних — `AccountDomainTests` та `UserDomainTests` тестують
чисту бізнес-логіку без інфраструктури. По-друге, доменні помилки
(`DomainError`) не знають про HTTP — контролер сам вирішує який статус
повернути на основі `error.Code`. По-третє, Rich Domain Model захищає
інваріанти на рівні сутності — неможливо змінити баланс рахунку обходячи
метод `Debit()` бо `Balance` має `private set`.

## Які недоліки з'явились?

Головний недолік — значно більше коду. З'явились окремі ORM entities
(`UserEntity`, `AccountEntity`) і маппери між ними та доменними моделями.
В лабі 1 одна модель виконувала обидві ролі. Навігація по проєкту
ускладнилась — щоб знайти де зберігається користувач треба пройти шлях
`AuthController` → `AuthService` → `IUserRepository` → `UserRepository`.

Також з'явилась проблема з `private set` в доменних моделях — EF Core
не може напряму встановити властивості при читанні з БД, тому потрібні
окремі ORM entities з публічними сеттерами і маппінг між ними.

## Наскільки простіше тепер змінити БД або фреймворк?

Щоб замінити PostgreSQL на MongoDB треба змінити тільки `MyBank.Infrastructure` —
реалізації репозиторіїв і контекст. `MyBank.Domain` і `MyBank.Application`
не зміняться взагалі. В лабі 1 треба було міняти сервіси, моделі і контекст.

Щоб замінити ASP.NET на інший фреймворк — міняється тільки `MyBank.Api`.
Вся бізнес-логіка залишається недоторканою.

## Чому обрано Rich Domain Model?

Обрано Rich Domain Model тому що вона краще захищає інваріанти.
В лабі 1 перевірка балансу знаходилась в `AccountService` — нічого
не заважало іншому сервісу змінити `Balance` напряму через `account.Balance = 0`.
В лабі 2 `Balance` має `private set` і змінюється тільки через `Debit()`
та `Credit()` які самі перевіряють всі правила і повертають `DomainError`
при порушенні. Логіка інкапсульована і не може бути обійдена.
